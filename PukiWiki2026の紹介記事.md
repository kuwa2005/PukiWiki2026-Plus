### 3.1 PukiWiki とは？

#### 書く内容

- PukiWiki（ぷきうぃき）は、**PHP で動く Wiki エンジン**（Web 上でページを追加・編集・リンクできる仕組み）であること。
- ページ名を `[[二重括弧]]` や WikiWord で書くとリンクになるなど、**日本語 Wiki として長年使われてきた**こと。
- テキストファイル（`wiki/*.txt`）にページを保存するシンプルな構成で、**データベース不要**で動くこと（一般的な説明として。DB モードの counter 等は詳細不要）。
- 管理者だけ編集可／誰でも編集可など、**運用形態を選べる**こと。

#### 必須ポイント

- 公式サイト: https://pukiwiki.osdn.jp/ （開発・DL: https://pukiwiki.osdn.jp/dev/）
- ライセンス: **GPL v2 or later**（README.txt より）
- 公式最新版（執筆時点の事実）: **PukiWiki 1.5.4**（2022-03-30 リリース。公式 FrontPage 等で確認可能）

#### 避けること

- 「日本唯一の Wiki」など根拠のない限定表現
- MediaWiki 等との優劣を断定的に書くこと
- PukiWiki Development Team の批判・人格攻撃

### 3.2 PukiWiki 1.5.4 UTF-8 と、メンテ停滞・セキュリティ課題

#### 書く内容

**事実として書けること:**

- 公式の最新安定版は **1.5.4**（2022-03-30）。1.5.3（2020-03-30）から約 2 年ぶりのリリースだった（報道・公式履歴より）。
- **2022-03-30 以降、公式に 1.5.5 等の新メジャー／マイナーリリースは確認できない**（執筆時点。執筆前に公式サイトで再確認すること）。
- 公式 README.txt の Copyright は **2001-2022**。
- 1.5.4 では **CVE-2022-36350 / CVE-2022-34486 / CVE-2022-27637** 等（1.5.3 以前向けの脆弱性）への対応が含まれる（SECURITY-AUDIT.md・NVD 参照）。
- 公式 1.5.4 の対応 PHP は **PHP 4.1〜8.1**（README.txt）。PHP 8.2 以降は公式確認範囲外であり、現代の PHP 環境では**動作確認・追加対策が利用者側の責任**になりうる。

**推測・解釈として書く場合（断定しない）:**

- 「長期間、公式のセキュリティアップデートが途絶えている**ように見える**」
- 「1.5.4 以降、新しい PHP バージョンや Web 脅威への公式追随が限定的**である可能性**」

**PukiWiki2026 の文脈:**

- 公式 1.5.4 UTF-8 をベースに、**認証必須・CSRF・静的セキュリティ監査対応**などを加えた非公式 fork である理由の背景として触れる。
- PukiWiki2026 自身も監査上 **未対応・部分対応の項目が残る**（HSTS/CSP 等の本番設定依存、NFKC 未適用等）。「完璧に安全」とは書かない。

#### 必須ポイント

- **事実と推測を分ける**（上記のとおり明示）
- 公式 PukiWiki に長年貢献してきた開発チームへの**敬意**を保つ
- 「公式は放置されている」「脆弱で使えない」等の**誹謗中傷・断定的批判は禁止**

#### 避けること

- 「公式 PukiWiki は死んだ／終わった」
- 開発者個人への攻撃
- CVE 番号の羅列だけで恐怖を煽る表現
- PukiWiki2026 が公式の全問題を解決したと読める表現

### 3.4 v1.0.0 / v1.0.1 の主な改善

#### 書く内容

### 3.5 導入の手軽さ — ZIP を置くだけで動く

#### 書く内容

1. GitHub **Release v1.0.1** から ZIP（Source code）をダウンロードし、解凍する。
2. Web サーバーのドキュメントルート（または vhost）に **`index.php` と `pukiwiki/` フォルダ一式**を配置する。
3. `pukiwiki/pukiwiki.ini.php.example` を **`pukiwiki/pukiwiki.ini.php` にコピー**して編集する（初回のみ）。
4. `pukiwiki/wiki/`・`cache/`・`backup/` 等に**書き込み権限**を付与（Unix/Linux 本番では起動時チェックあり）。
5. ブラウザで **`index.php`** にアクセス → 初期ページ表示を確認。
6. **初回ログイン:** ユーザー名 `editor` / パスワード `editor`（**デモ用・必ず変更**）→ パスワード変更画面が表示され、変更完了まで他操作不可（v1.0.1）。

**手軽さのポイント（記事で強調してよい）:**

- DB セットアップ不要（標準構成）
- Composer 等のビルドステップ不要
- PHP ビルトインサーバーでも動作確認可能（`php -S localhost:8080`）
- バックアップは **`index.php` + `pukiwiki/` をコピー**するだけ

**要件（簡潔に）:**

- PHP **8.1 以上推奨**（CI は 8.1〜8.4）
- Apache 2.4 または nginx
- 拡張: mbstring, session, json 等

#### 必須ポイント

- **初期パスワード変更必須**を必ず警告として入れる
- 「解凍して置くだけ」は **ini コピー・権限・初回パスワード変更**まで含めた「比較的手軽」と表現してよい

#### 避けること

- 「インストーラー不要＝設定一切不要」と書くこと
- 本番では `tools/gen-password-hash.php` を公開したままにするよう促すこと（**削除または IP 制限推奨**）

### 3.6 誰向けか / 向かないケース

#### 書く内容

**向いている例:**

- 小規模な社内 Wiki・個人メモ・趣味サイトで **PukiWiki 記法をそのまま使いたい**人
- 公式 1.5.4 をベースに、**匿名編集を止めて認証必須**にしたい人
- PHP 共有レンタルサーバー等で **軽量 Wiki** を試したい人
- セキュリティ監査結果を公開している fork を**検証環境で評価**したい技術者

**向かない／注意が必要な例:**

- **公式 PukiWiki との完全互換・公式サポート**が必要な場合（非公式 fork のため）
- 大規模マルチユーザー・細かい ACL が必要な場合（PukiWiki 標準の認証モデルの範囲内）
- **PHP 8.1 未満**やレガシー環境のみの場合（推奨外）
- 監査未対応項目（HTTP セキュリティヘッダー、NFKC 等）を**ゼロリスク**として扱いたい厳格な要件

#### 必須ポイント

- メリットだけでなく**限界**も公平に
- 「非公式」であることとの整合

#### 避けること

- エンタープライズ向け CMS の代替と断定すること

## 5. 盛り込まない・注意事項

### 公式 PukiWiki との関係

- PukiWiki2026 は **PukiWiki Development Team による公式リリースではない**。
- 「公式」「正規版」「後継版」等の表現は**使用禁止**。
- 公式の名称・ロゴを、公式と誤認される使い方をしない。

### ライセンス（GPL）

- GPL v2 or later に基づく fork である。
- 配布・公開時の **ソース開示義務** がある（詳細は GPL 条文・`COPYING.txt` 参照。記事では「GPL に留意」と程度で可）。

### 非公式 fork の限界

- 公式サポート窓口はない。Issue は GitHub のみ。
- 上流 1.5.5 等への追随は**保証されない**（`UPSTREAM.md` 参照）。
- セキュリティ監査は**静的コードレビュー**であり、ペネトレーションテスト完了を意味しない。

### 誇大表現の禁止

| NG 例 | OK 例 |
|-------|-------|
| 完全に安全な Wiki | セキュリティ強化を目的とした fork |
| 公式 PukiWiki の問題をすべて解決 | 1.5.4 ベースに CSRF・認証必須等を追加 |
| 誰でも本番ですぐ使える | 検証後、パスワード変更等のセットアップを経て利用可能 |
| 公式は脆弱で使えない | 1.5.4 以降、公式の新リリースは確認できない（執筆時点） |

### その他

- `.env` や本番パスワードの具体値を記事に載せない。
- v1.0.0 の初期パスワード `pass` と v1.0.1 の `editor` を混同しない（**記事は v1.0.1 基準**）。
- kuwa2005 個人の私的生活や推測による動機は書かない。

### 案 A（背景から入る）

> 日本語 Wiki といえば PukiWiki を思い浮かべる人も多いでしょう。シンプルなファイルベースの構造と、なじみ深い記法は今も魅力です。一方で、公式の最新版 1.5.4 は 2022 年にリリースされた後、執筆時点では後続のバージョンアップが確認できません。PHP の世代も進み、公開 Wiki では認証や CSRF 対策をより意識する時代になりました。そこで注目したいのが、1.5.4 UTF-8 をベースにセキュリティ強化を加えた非公式 fork「PukiWiki2026」です。GitHub から ZIP を取得し、サーバーに置いて設定ファイルをコピーするだけで、比較的短い手順で試せます。本記事では、PukiWiki の位置づけから、PukiWiki2026 の改善点・導入の流れまでを整理します。

### 案 B（課題→解決の構成）

> 「PukiWiki を今の PHP 環境で動かしたい。でも匿名編集のまま公開するのは不安だ」——そんな声に応える選択肢の一つが PukiWiki2026 です。PukiWiki 1.5.4 UTF-8 を土台に、編集の認証必須化、CSRF トークン、スパム対策のオプション、oEmbed などを載せた非公式 fork です。v1.0.1 ではディレクトリ構成の整理や Unicode まわりの hardening も加わりました。公式の配布物ではない点は理解したうえで、Release から落とした ZIP を Web サーバーに展開し、初期ユーザー `editor` のパスワードを変更すれば、手早く動作確認できます。

### 案 C（技術メディア向け・短め）

> PukiWiki2026 v1.0.1 は、PukiWiki 1.5.4 UTF-8 向けのセキュリティ強化 fork です。編集認証の既定 ON、CSRF、2026 年時点の静的監査、PHP 8.1〜8.4 の CI を特徴とし、構成は `index.php` と `pukiwiki/` ディレクトリに集約されています。Composer も DB も不要で、レンタルサーバーへの持ち込みが容易です。本記事では公式版との関係、v1.0.0/1 の差分、セットアップ手順を事実ベースで紹介します。
